# SCMDB Log Watcher

**TL;DR — for impatient users:**

1. Install [Python from the Microsoft Store](https://apps.microsoft.com/detail/9pnrbtzxmb4z) (one click).
2. Unpack this ZIP somewhere (e.g. on your Desktop).
3. Double-click **`install.bat`** once.
4. Double-click **`run-watcher.bat`** before or after launching Star Citizen.
5. Open [scmdb.net](https://scmdb.net) — the page detects the watcher automatically.

Done. The GUI can run in system tray while you play.

The Python implementation is the only maintained production app in this repository.

Linux users with Star Citizen via Proton/Wine: see

## What it does

A small Python script that tails Star Citizen's `Game.log`, parses mission and
blueprint events, and pushes them live to an open [SCMDB](https://scmdb.net)
browser tab via Server-Sent Events:

- Mission timers start and stop automatically when you accept / complete a contract.
- Completed runs open the Community-Run dialog pre-filled with mission and elapsed time.
- Blueprint drops are matched against the mission that just finished.

The script runs on `127.0.0.1:23456` only — no LAN exposure, no telemetry,
no external network access. CORS is strictly whitelisted to scmdb.net.

Design rationale, log patterns, CORS strategy and all open decisions live in
[`gamelog-watcher-concept.md`](https://github.com/KrovaxCode/SCMDB_LOG_WATCHER/blob/main/gamelog-watcher-concept.md)
on GitHub.

## Why the Python implementation remains the release target

- **Single maintenance path** — UI, runtime, import, installer and translations only need to be updated once.
- **Windows packaging already solved** — the current build produces an EXE and installer ready for Program Files deployment.
- **Portable source base** — Star Citizen under Proton/Wine on Linux still works from source.
- **User-editable translations** — installed language files can be edited without recompiling.

## Install on Windows (one-time)

1. **Install Python 3.10 or newer.** Easiest path on Windows:
   [Python from the Microsoft Store](https://apps.microsoft.com/detail/9pnrbtzxmb4z)
   (one click, adds itself to PATH). Or grab the installer at
   [python.org/downloads](https://www.python.org/downloads/) and tick
   "Add Python to PATH".

2. **Get the watcher.** Two ways:
   - **ZIP download (recommended for most users):** grab the latest
     `SCMDB_LOG_WATCHER_v<version>.zip` from
     [scmdb.net](https://scmdb.net) and unpack it.
   - **Git clone (if you know what that means):**

     ```bash
     git clone https://github.com/KrovaxCode/SCMDB_LOG_WATCHER.git
     ```

3. **For source-tree setup only, run `install.bat`** (double-click). This is a
   developer bootstrap that checks Python prerequisites, creates a local `.venv`
   and installs dependencies so you can work from the repository.

   End users should not use `install.bat`; they should run the final installer
   generated by Inno Setup instead.

## Install and run on Linux

Star Citizen via Proton/Wine writes the same `Game.log`, so the watcher
works identically on Linux — just without the Windows batch launchers.
Pretty straightforward:

1. Unzip the watcher where you want and open a terminal in that directory.
1. Create a virtual environment:

  ```bash
  python -m venv .venv
  ```

1. Activate it (use the file matching your shell — `activate` for bash/zsh,
   `activate.fish`, `activate.csh`, …):

  ```bash
  source .venv/bin/activate
  ```

1. Install Flask in the virtual environment:

  ```bash
  python -m pip install flask
  ```

To run the live watcher:

```bash
python watcher.py --log-path /path/to/your/Game.log
```

To import previous sessions:

```bash
python watcher.py import --logbackups-dir /path/to/your/logbackups
```

You have to activate the virtual environment (step 3) in each new terminal
before running these commands.

That's all folks! ;)

*Thanks to **Meliok** for contributing these Linux instructions.*

## Run

Double-click **`run-watcher.bat`**. It launches the GUI (headless runtime for the
watcher process, no persistent terminal required).

Then open [scmdb.net](https://scmdb.net) — the page detects the watcher
automatically and shows a live indicator.

### GUI mode (default on Windows)

`run-watcher.bat` now launches a small GUI (no persistent terminal window):

- Installation values are stored in `runtime/watcher-config.json` and can be edited later
  via **Settings**.
- It starts the watcher in the background and shows a red/green status
  indicator (`red = disconnected/stopped`, `green = connected/running`).
- It includes buttons to **Disconnect** (stop the watcher process) and
  **Import** (run historical import and write JSON to `runtime/scmdb-import/`).
- The GUI is single-instance. If already open (including tray mode), launching
  it again exits immediately.
- If enabled in the GUI, it auto-starts the watcher when `StarCitizen.exe` is
  detected.
- Closing the window sends it to the system tray (if tray dependencies are
  installed). Tray icon color mirrors state (`red/orange/green`).
- The language selector is GUI-only and does not affect log parsing.
- UI translations are loaded from the `lang/` folder next to the executable.
- Installed builds keep those files under `Program Files\SCMDB Log Watcher\lang` so users can edit or add their own translations.
- Built-in language files now include `en-en`, `es-es`, `fr-fr`, `de-de`, `it-it`, `pt-pt`, `ru-ru`, `zh-cn` and `ja-jp`.
- On startup, GUI checks release feed and shows whether a newer version exists.
- Failsafe: before starting a new watcher instance, GUI closes orphan `watcher.py`
  processes to recover from previous inconsistent launches.

## Packaging and installer build

A dedicated folder contains scripts and instructions:

- `installer/build-app-exe.bat`
- `installer/build-installer-exe.bat`
- `installer/INNO_SETUP_TEMPLATE.iss`
- `installer/README.md`

Use those scripts from VS Code terminal to rebuild executable artifacts after
code changes. The app build now produces a single `dist/SCMDB-Watcher.exe`
that launches the GUI by default and switches to live/import mode through
internal subcommands.

The release artifact users should download is the single Inno Setup package:

`release\SCMDB-Log-Watcher-Setup.exe`

The installer requests admin elevation so the app lands under Program Files,
while runtime data still stays in LocalAppData per user.

The build scripts also mirror the editable language files into:

`release\py\lang\`

so the release tree matches the installed layout.

Advanced users can still run the watcher directly from a terminal:

```powershell
.venv\Scripts\python.exe watcher.py --log-path "C:\...\Game.log"
```

### Command-line options

```text
run-watcher.bat [--port PORT] [--log-path PATH] [--dev] [--dev-origins URL,URL] [-v]
```

| Flag | Default | Description |
| --- | --- | --- |
| `--port` | `23456` | TCP port to bind on `127.0.0.1` |
| `--log-path` | `C:\Program Files\Roberts Space Industries\StarCitizen\LIVE\Game.log` | Path to `Game.log`. Override for PTU/HOTFIX/EPTU or non-default installs. |
| `--dev` | off | Also allow `http://localhost:5173` and `:3000` as CORS origins (for local SCMDB development) |
| `--dev-origins` | — | Comma-separated list of extra allowed origins (implies `--dev`) |
| `-v` / `--verbose` | off | Debug-level logging |

All flags can also be set via environment variable: `SCMDB_WATCHER_PORT`,
`SCMDB_WATCHER_LOG_PATH`, `SCMDB_WATCHER_DEV=1`, `SCMDB_WATCHER_DEV_ORIGINS`.

### Example: tracking PTU

```powershell
run-watcher.bat --log-path "C:\Program Files\Roberts Space Industries\StarCitizen\PTU\Game.log"
```

### Example: custom port (if 23456 is taken)

```powershell
run-watcher.bat --port 34567
```

In SCMDB, set the matching port:

```js
localStorage.setItem('scmdbWatcherPort', '34567')
```

…and reload the tab.

## Import historical missions (one-shot)

If you want to backfill all your past completed missions and blueprint drops
into SCMDB, double-click **`run-import.bat`**. It scans every backup log file
Star Citizen has kept (`<game>\<channel>\logbackups\`) and writes a single
JSON file you can hand to SCMDB:

```text
runtime/scmdb-import/scmdb-import-2026-05-13_07-31-30.json
```

The path is printed when the import finishes. Only successfully completed
missions (`CompletionType[Complete]`) and blueprint drops that can be
correlated to such a completed mission are exported — abandoned, failed
or disconnected runs are intentionally skipped because they have no
meaningful run time for community-run statistics. Mission GUIDs are
de-duplicated within the export and the SCMDB backend can dedupe again
against runs you've already submitted live.

Defaults are wired up for the LIVE installation. To import from PTU/HOTFIX
instead:

```powershell
run-import.bat --log-path "C:\Program Files\Roberts Space Industries\StarCitizen\PTU\Game.log"
```

To also include the *current* (still-being-written) `Game.log` — useful if
you forgot to start the live watcher and want to capture today's runs:

```powershell
run-import.bat --include-current
```

## Session log files

Every time you start the watcher, a fresh log file is created under `runtime/logs/`:

```text
runtime/logs/watcher-2026-05-13_00-12-34.log
```

It contains everything the console window printed, with full date+time
prefixes. If something looks off and you want to file a bug, attach the
matching log file from your session — that gives the maintainer the exact
event sequence to reproduce against.

The `runtime/logs/` directory is never auto-cleaned. Delete old files manually
when you don't need them anymore.

## What the watcher emits

The watcher exposes three endpoints on `127.0.0.1:<port>`:

- `GET /ping` — returns `{"status":"ok","version":"<x.y.z>"}`. SCMDB uses
  this to detect whether the watcher is running and whether it's up to date.
- `GET /state` — JSON snapshot of currently active missions.
- `GET /events` — Server-Sent Events stream. Each event is a JSON object:

| `type` | Payload |
| --- | --- |
| `state_snapshot` | `{ active: [...] }` sent once to every new subscriber |
| `mission_start` | `{ guid, debugName, generator, startTs }` |
| `mission_complete` | `{ guid, debugName, generator, completion: "Complete", reason, endTs }` |
| `mission_ended` | `{ guid, debugName, generator, completion, reason, endTs }` for `Abandon`/`Fail`/`Disconnect`/`Deactivate` |
| `blueprint_received` | `{ productName, missionGuid, missionDebugName, missionTrigger, ts }` — `productName` is the full localized item name from the in-game notification (e.g. `"Antium Helmet Maroon"`, `"R97 Shotgun"`). `missionTrigger` is `"complete"` (typical: blueprint awarded at mission completion) or `"accept"` (some crafting contracts award the blueprint when accepted). `null` if no recent mission could be correlated. |
| `session_reset` | `{}` — emitted when SC restart is detected (log rotation) |

## Versioning

The current version is set at the top of `watcher.py` as `__version__`. It
appears in the startup banner, in every session log file, and in the
`/ping` response. The version on the SCMDB download page should match
the latest GitHub tag.

## Troubleshooting

### Watcher console shows "Game.log not found, waiting…"

You're either not running Star Citizen yet (fine — it'll attach as soon as
SC writes the file), or your install path differs from the default. Pass
`--log-path` accordingly.

### "Failed to bind 127.0.0.1:23456"

Some other application is using the port. Try `--port 34567`.

### SCMDB tab does not detect the watcher

- Make sure the console window is still open and shows
  `Listening on http://127.0.0.1:23456`.
- Confirm in the browser DevTools Network tab that the `ping` request
  succeeds (status 200) and the response Origin header matches your tab's
  origin.
- If you're running SCMDB locally on a non-standard port, start the watcher
  with `--dev` or `--dev-origins`.

### A mission was missed because I started the watcher after SC

The watcher reads `Game.log` from the start of the file, so any mission
**still active** when the watcher starts is recovered with the correct start
time. Missions already completed before watcher start are intentionally not
replayed (no ghost dialogs).
